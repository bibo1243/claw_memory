# 🤝 Multi-Agent File Collaboration (多智能體協作編輯)

## 📌 Context

當 Gary 同時使用多台 AI 實例（如 Trae + Antigravity）針對同一個主題作業時，必須有明確的「分工協議」，避免：
- 兩台 Agent 同時修改同一份檔案，導致後寫的覆蓋前寫的
- 跨議題邏輯不一致（各自獨立寫，口徑矛盾）
- 重複研究相同資料，浪費算力

## 🎯 核心原則

### 原則一：一份檔案，一台 Agent
> ⛔ **絕對禁止兩台 Agent 同時修改同一份檔案**

同一時間只能有一台 Agent 對特定檔案進行寫入操作。若需要多台 Agent 的產出，採用「分檔作業 → 合併」策略。

### 原則二：分檔作業 → 最後合併

**流程圖：**
```
                    ┌──────────────────┐
                    │   原始主檔（凍結） │
                    │   ⛔ 雙方不動     │
                    └────────┬─────────┘
                             │
                 ┌───────────┴───────────┐
                 │                       │
          ┌──────▼──────┐         ┌──────▼──────┐
          │  Agent A     │         │  Agent B     │
          │  分支檔案 A  │         │  分支檔案 B  │
          └──────┬──────┘         └──────┬──────┘
                 │                       │
                 └───────────┬───────────┘
                             │
                    ┌────────▼─────────┐
                    │  合併 Agent       │
                    │  (指定一台負責)    │
                    │  → 產出最終版     │
                    └──────────────────┘
```

### 原則三：透過共享記憶協調

所有分工指令和任務狀態都要寫入共享記憶 (`/Users/leegary/個人app/Inbox/共享記憶.md`)，讓兩台 Agent 都能看到。

## 📋 標準作業流程 (SOP)

### Step 1：分析議題獨立性

將要處理的內容按「獨立性」分組：
- **完全獨立**：兩個議題之間沒有因果關係 → 可分給不同 Agent
- **有交叉引用**：議題 A 的結論會影響議題 B → 應由同一台 Agent 處理，或先做完 A 再做 B
- **高度耦合**：多個議題互相影響 → 由一台 Agent 統一處理

### Step 2：建立分支檔案

命名規則：
```
{原始檔名}_分支_{Agent名稱或議題名}.md
```

範例：
```
📁 Inbox/
├── 115.03.09_單位主管會議議題檢視.md          ← 原始主檔（凍結）
├── 115.03.09_會議準備_行政組議題.md            ← Agent A 負責
└── 115.03.09_會議準備_圖教社資討論事項.md       ← Agent B 負責
```

### Step 3：寫入共享記憶

在 `共享記憶.md` 的訊息佇列寫入分工指令：

```markdown
## 📮 協作任務 [日期: YYYY-MM-DD]

### 任務名稱
[任務描述]

### 分工表
| 負責 Agent | 負責議題 | 輸出檔案 | 狀態 |
|-----------|---------|---------|------|
| Agent A   | 議題1, 議題2 | xxx_分支_A.md | 🟡 進行中 |
| Agent B   | 議題3, 議題4 | xxx_分支_B.md | ⏳ 待開始 |

### 協作規則
- ⛔ 禁止直接修改原始檔 `xxx.md`
- ✅ 各自在分支檔案中作業
- 🔄 完成後由 [指定 Agent] 統一合併
```

### Step 4：合併與交付

由指定的 Agent（通常是先完成的那一台）：
1. 讀取所有分支檔案
2. 檢查邏輯一致性（口徑、語氣、引用數據是否矛盾）
3. 合併回主檔或產出新的最終版
4. 更新共享記憶中的任務狀態為 ✅ 已完成

## ⚠️ 衝突類型與預防

| 衝突類型 | 風險等級 | 預防措施 |
|----------|---------|---------|
| **同時寫入同一檔案** | 🔴 高 | 嚴格執行「一檔一 Agent」原則 |
| **跨議題邏輯不一致** | 🟡 中 | 有交叉引用的議題由同一台 Agent 處理 |
| **重複研究** | 🟢 低 | 透過共享記憶分配資料來源 |
| **合併遺漏** | 🟡 中 | 合併時逐項 checklist 核對 |

## 🔧 適用場景

1. **會議準備**：多個議題分配給不同 Agent 研究
2. **文件修訂**：不同章節分配給不同 Agent 改寫
3. **資料研究**：不同資料來源分配給不同 Agent 查閱
4. **專案規劃**：不同模組分配給不同 Agent 設計
