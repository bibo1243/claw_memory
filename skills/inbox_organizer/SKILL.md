# Inbox Organizer (收件匣自動分類助手)

## 目標 (Goal)
幫助使用者有系統地管理放置於 `/Users/leegary/個人app/Inbox` 中的零碎檔案與資料，將其自動歸類到相應的工作區或個人知識庫 (Personal Knowledge Management system) 資料夾。

## 執行流程 (Workflow)
當使用者要求「inbox」、「歸類」「幫我整理 Inbox」、「處理 Inbox」或我發現有新資料需要分類時，請執行以下步驟：

1. **讀取 Inbox**：
   - 掃描 `/Users/leegary/個人app/Inbox` 內的所有檔案。
   - 根據檔名、副檔名與簡單的內容分析判斷其所屬分類。如為圖片截圖，可運用內建視覺識別(Vision/OCR)檢視內容關鍵字。
2. **分析與分配**：
   - 先檢查在 `/Users/leegary/個人app/個人資料夾` 底下是否已有適合的目錄，判斷該檔案是否屬於其中。
   - **特定自動分類規則**：
     - **【放行記錄】**：
       - 若截圖或文件內容出現「收款人資料」、「付款金額」、「手續費負擔別」、「退件原因」等匯款放行特徵，一律自動歸類並移至 `/Users/leegary/個人app/個人資料夾/放行記錄/`，並在必要時依日期重命名（如 `放行記錄_YYYYMMDD.png`）。
       - **Notion 紀錄更新排版規範 (⚠️強制執行)**：在寫入 Notion (Gaa/Task 或專案) 時，**必須**使用 Toggle (折疊) 區塊進行排版。Toggle 標題須以「YYYY-MM-DD 放行記錄與明細」為開頭，並將「OCR 分析之匯款明細」與「Google Drive 雲端檔案連結」皆收攏在 Toggle 區塊之內，避免版面愈來愈長。
     - **【會議記錄】**：
       - 歸檔路徑必須採用 **兩層結構**：`會議記錄/{會議系列名稱}/{YYYYMMDD+場次名}/`
       - **第一層**：會議系列的統稱（例如 `慈光核心主管會議`、`組織發展會議`、`機構主管會議`、`董事會`）。
       - **第二層**：該次會議的日期與場次名（例如 `20260226核心主管會議`）。
       - 同一場會議的所有相關檔案（議程、紀錄、逐字稿、附件等）放在同一個日期場次資料夾內。
       - 範例：`個人資料夾/會議記錄/慈光核心主管會議/20260226核心主管會議/`
     - **【專案版本控管】**：
       - 若同一專案（如人資系統計畫書）有多個版本或不同日期的檔案，採用 **「當前唯一 + 單一封存槽」** 策略。
       - **根目錄永遠只放「最新的一份」**：在專案根目錄只保留最新版本，並可在檔名前標註 `[最新]`。
       - **舊版本統一進入「歷史版本」**：在專案根目錄下建立唯一的 `歷史版本` (或 `Archive_舊版計畫`) 資料夾，當新版本出現時，自動將舊版本移入歷史區備查，確保專案根目錄視覺乾淨。
   - 若不屬於現有專案，根據檔案性質（例如財務、學習、靈感碎片、收據等）提議為其建立新目錄，這也**必須在 `個人資料夾` 之內**（例如 `/Users/leegary/個人app/個人資料夾/個人財務`, `/Users/leegary/個人app/個人資料夾/閱讀筆記`）。
   - **絕對不要**將檔案搬移到同階層的其他目錄（如 `我的日記軟件`, `年度目標` 等）。
3. **提交計畫與確認（⚠️ 強制遵循 Safe Delete Policy）**：
   - 列出即將發生的變動計畫（搬移來源、目的路徑）。
   - 例如：
     - `Inbox/2026預算草稿.xlsx` -> 移至 `/Users/leegary/個人app/個人資料夾/個人財務/2026預算草稿.xlsx`
     - `Inbox/新的好點子.txt` -> 移至 `/Users/leegary/個人app/個人資料夾/草稿/新的好點子.txt`
   - **明確詢問使用者**：「這是我建議的整理與移動計畫，請問是否同意執行？(Y/n)」
4. **執行移動與雲端上傳 (取得連結)**：
   - 等待使用者「同意」後。
   - 若有需要，先建立缺少的分類資料夾 (`mkdir -p`)。
   - 使用指令移動檔案至指定目錄。
   - ⚠️ **立即上傳以取得連結**：為了避免等待背景雙向同步工具的時間，於本地搬移後，立即幫忙執行單檔加速上傳工具：
     `/usr/bin/python3 /Users/leegary/個人app/scripts/upload_to_gdrive.py "搬移後之絕對路徑"`
   - 該腳本將立即返回 Google Drive 連結，請將取得的連結一併回報給使用者，以利使用者迅速投入後續工作。
   - *(註：本地雙向同步腳本運行時若偵測到該檔案在雲端已存在，將自動執行合併與更新，絕不會產生重複上傳的衝突，此法既快又安全。)*

## 備註
- 如果對某些檔案完全沒有分類靈感，可以將其分類到 `/Users/leegary/個人app/個人資料夾/Archive/未分類`，或者直接詢問使用者該如何定義這類檔案。

## 雲端雙向同步機制 (Google Drive Mirror Sync)
針對 `/Users/leegary/個人app/個人資料夾` 的內容，系統中已建立一套即時的 Google Drive 鏡像同步機制：
- **同步目標**: Google Drive 的 `gaa_all/上傳/個人資料夾` (使用 Drive REST API 進行同步)。
- **運作原理**:
  1. 系統背景運行一個 `LaunchAgent` 服務 (`com.leegary.sync-personal-folder`)。
  2. 該服務利用 `fswatch` 即時監聽 `個人資料夾` 下的任何變動 (新增、修改、刪除、移動)。
  3. 當偵測到檔案變動後，會在延遲合併事件後，觸發 Python 腳本 (`scripts/sync-personal-folder.py`)。
  4. 腳本會精準比對本地端與雲端資料夾內容：
     - 將新增或更新的檔案上傳/覆寫至雲端。
     - 若本地端檔案已被移動或刪除，同步將雲端對應之舊檔案移入「垃圾桶 (Trashed)」。
- **手動觸發**: 如果遇到未同步的情況，可以手動在終端機執行 `/usr/bin/python3 /Users/leegary/個人app/scripts/sync-personal-folder.py sync` 來強制啟動一次完整鏡像比對。
